# Good luck Low-Touch Operating Mode

Updated: 2026-06-13

Purpose: reduce the user's time cost. Codex should do screening, reading, drafting, and preparation. The user should only be asked for short approvals at real decision points.

## Core Rule

The user is not the task screener.

Codex should not ask the user to inspect every listing, read every page, or judge every requirement. Codex should inspect listings first and only bring forward candidates that pass the safety and effort filters.

## User Time Budget

Target user involvement:

- Normal day: 0-5 minutes
- Submission day: 5-10 minutes
- Payment / wallet / account day: user approval required, no time estimate

## What Codex Can Do Alone

- Browse public task listings.
- Read task details.
- Reject unsafe or unsuitable tasks.
- Compare effort versus likely reward.
- Draft X threads, articles, documentation reviews, and reports.
- Create local files and portfolio samples.
- Prepare GitHub issues, comments, or Superteam submissions up to the final confirmation step.
- Track opportunities in `opportunity-log.md`.

## When Codex Must Ask the User

Ask only when the next step would do one of these:

- Create or change an external account.
- Submit a task to a third-party site.
- Post from the user's X / GitHub / Telegram / email account.
- Connect a wallet.
- Sign any wallet message.
- Transfer funds.
- Deposit, stake, bridge, borrow, lend, or trade.
- Share personal contact, Telegram, phone, bank, payout, or identity information.
- Use Payoneer.

Payoneer reminder:

- Do not open Payoneer on the computer.
- Use only the user's phone with the Macau SIM network if Payoneer must be checked.

## Automatic Reject Rules

Reject a task immediately if it requires:

- Deposit / recharge / top up / buy credits.
- Stake / lock tokens.
- Borrow / lend / run DeFi positions.
- Real trading or prediction market participation.
- Wallet connection before the task can be understood.
- Wallet signature.
- KYC or passport submission.
- Downloading unknown software.
- Public personal video or face/voice content unless the user explicitly wants it.
- Offline or in-person event attendance.
- Region eligibility that does not match the user.
- Winner-takes-all with many submissions, unless the task is extremely easy and useful as practice.

## Preferred Task Types

Best:

- Documentation cleanup.
- Beginner onboarding review.
- Product explanation article.
- Research summary.
- X thread that can be written from public information.
- Product feedback based on public pages or no-login demos.

Acceptable:

- Small GitHub issue / PR if no private setup is required.
- Small visual/content task if the output can be generated safely.

Avoid for now:

- DeFi testing.
- Prediction markets.
- Gaming tasks requiring payment.
- Offline event content.
- Video tasks requiring user appearance.
- Heavy development bounties.

## Candidate Report Format

When Codex finds a possible task, report only this:

- Task:
- Link:
- Why it may fit:
- Reward:
- Time estimate:
- User action needed:
- Risk check:
- Recommendation: do / skip / watch

Do not ask the user to inspect the full page unless there is no other way.

## Submission Rule

Codex can prepare everything.

Before the final external submission, Codex must ask:

`确认提交到 [site/project] 吗？内容是 [short summary]。`

Only submit after the user clearly confirms.
